# CURRENT_DIRECTION.md

## 当前方向

当前优先级最高的目标，已经不再是继续围绕“代理排序 / trust score / verify / refresh 收口”做局部优化，而是把项目主线正式上提到：

> **构建稳定、可持续、可解释的真实 browser 执行系统。**

这意味着后续工作要从“选前判断是否更准”升级到“真实执行是否稳定、身份是否一致、结果是否可运营”。

1. 先把 **真实 browser 执行链** 做稳，让任务在真实执行路径上持续可跑、可复现、可定位问题
2. 把 **proxy + fingerprint + session identity** 统一成执行身份模型，避免网络身份、设备身份、会话身份各自为政
3. 让 **verify / trust score** 从“选前判断”升级为“执行前 + 执行中 + 执行后”的闭环质量系统
4. 把 **status / explain / result** 从调试视角升级为长期运营级控制面
5. 把 **性能与并发治理** 作为支撑能力持续推进，但不再把它写成阶段总纲本身

## 当前判断

项目已经明显脱离“继续补最小骨架”与“继续打磨单点排序”的阶段。

当前系统已经具备：
- 浏览器执行系统 V1
- Fingerprint profile 注入第一版
- 代理池基础能力
- sticky session 正式绑定
- smoke / verify / batch verify / 巡检 V1
- 代理选择策略层第一版
- trust score 起点与主链接入
- explain / status / result 基础可观测面

因此，当前阶段的主要方向是：

> **从“代理选择系统收口”升级到“真实 browser 执行主线稳定化”，让身份、质量判断、执行结果、控制面开始围绕真实执行闭环统一。**

补充方向：允许未来增强 HTML/debug/trace retention，但不因此提升 screenshot / GUI / 重视觉能力的优先级；更大磁盘也不改变“真实 browser 执行主线优先、能力长期可运营”的排序。

## 本阶段重点

### 主线 A：真实 browser 执行链稳定化
- 让真实浏览器任务链路更稳定：创建、调度、执行、结果回写、失败定位形成可持续闭环
- 优先解决真实执行中的不稳定点，而不是继续把重点放在选前排序细节上
- 让执行链具备长期跑批、持续回归、真实任务验证能力

### 主线 B：统一执行身份模型
- 把 proxy、fingerprint、session 绑定关系升级为统一执行身份表达
- 让网络出口、设备指纹、地区一致性、会话连续性在同一模型里被消费和解释
- 避免 selection、verify、runtime、result 各自维护不同身份语义

### 主线 C：verify / trust score 升级为执行闭环
- 保留 verify / trust score 在选前判断中的作用，但不再把它们视为主线本身
- 推进它们进入执行前、执行中、执行后的统一质量闭环
- 让 verify 结果、运行结果、失败类型、会话表现一起反哺真实执行质量判断

### 主线 D：长期运营级控制面
- 把 status / explain / result 从“调试能看”升级到“长期运营可用”
- 让系统能够回答：当前执行是否稳定、身份是否一致、失败集中在哪、哪些质量信号正在恶化
- 让控制面服务于持续运营，而不只是一次性排查

### 主线 E：性能与并发治理作为支撑
- 持续控制高并发下的数据库写放大、claim/reclaim 抖动与状态竞争
- 继续补齐 metrics、profiling、状态聚合效率与写回成本治理
- 目标是支撑真实 browser 执行主线长期运行，而不是把性能治理单独写成阶段总纲

## 当前阶段定义

当前阶段不是“继续小修小补代理排序”，而是：

> **围绕真实 browser 执行，完成一次阶段升级：把执行链稳定化、执行身份统一化、质量判断闭环化、控制面运营化。**

补充约束：verify、trust score、refresh、并发治理仍然重要，但它们现在的角色是服务真实 browser 执行主线，而不是继续单独充当项目总方向。
